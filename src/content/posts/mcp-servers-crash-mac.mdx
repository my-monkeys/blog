## Le symptôme

Mon Mac crash. Encore. Pour la troisième fois en une heure.

Pas un kernel panic propre — le genre de crash mou où macOS commence par tuer Safari, puis le Finder, puis se fige complètement. Le ventilateur hurle. Force reboot, ça repart 10 minutes, ça recrashe.

J'ouvre un terminal et je regarde les basics :

```bash
ps aux -m | awk 'NR>1 {sum+=$6} END {printf "RAM: %.1f GB\n", sum/1024/1024}'
# RAM: 18.2 GB

sysctl vm.swapusage
# total = 3072.00M  used = 1868.44M  free = 1203.56M
```

18 GB de RSS sur une machine qui en a 16. Presque 2 GB de swap. macOS compresse en mémoire ce qu'il peut, mais il n'y a plus rien à compresser.

## L'enquête

Premier réflexe : qu'est-ce qui mange autant ?

```bash
ps aux | grep 'claude' | grep -v grep | wc -l
# 10
```

10 processus Claude. J'en ai ouvert **un seul** terminal.

```bash
ps aux | grep -E 'npm exec|mcp' | grep -v grep | wc -l
# 50
```

50 serveurs MCP. 50.

## Le coupable

En détaillant les processus Claude, le pattern saute aux yeux :

```bash
PID 2768   claude --dangerously-skip-permissions --resume   # ← ma session
PID 6634   claude -p "You are analyzing a person's Claude Code brain..."
PID 10440  claude -p "You are analyzing a person's Claude Code brain..."
PID 17129  claude -p "You are analyzing a person's Claude Code brain..."
PID 21303  claude -p "You are analyzing a person's Claude Code brain..."
PID 25119  claude -p "You are analyzing a person's Claude Code brain..."
PID 28953  claude -p "You are analyzing a person's Claude Code brain..."
```

6 instances `claude -p` identiques lancées par le plugin **claude-brain-sync**. Chacune est un subagent `evolve --auto` qui "analyse le cerveau pour promouvoir des patterns" — une sorte de maintenance automatique de la mémoire.

En parallèle, des dizaines de scripts bash orphelins :

Le problème est double :

1. **Chaque subagent `claude -p` charge tous les MCP servers configurés globalement** — y compris ceux dont il n'a absolument pas besoin (Dart LSP, iOS Simulator, Minecraft schematic builder... pour "analyser un brain"). Sur ma config, ça fait ~25 MCP servers, soit **~2 GB par subagent**.

2. **Les processus survivent au crash de la session parente.** Quand macOS force-kill ma session Claude sous pression mémoire, les subagents et leurs MCP restent orphelins. Au reboot de la session, le hook relance un nouveau batch. Les anciens sont toujours là. En 3-4 crashs, on arrive à 50+ processus.

## Le fix

Étape 1 — kill immédiat :

```bash
ps aux | grep 'claude-brain-sync\|claude -p You' | grep -v grep \
  | awk '{print $2}' | xargs kill
```

Résultat instantané :

## La leçon

<Callout type="warn">
Les plugins Claude Code qui utilisent des hooks `session_start` pour spawner des subagents (`claude -p`) sont une bombe à retardement sur les machines avec beaucoup de MCP servers configurés globalement. Chaque subagent hérite de **tous** les MCP servers, même s'il n'en utilise aucun.
</Callout>

Trois choses à retenir :

**1. Surveille le nombre de processus Claude.** Si `ps aux | grep claude | wc -l` retourne plus que le nombre de terminaux ouverts, il y a un problème.

**2. Déplace les MCP spécialisés du global vers le projet.** Dart LSP n'a rien à faire dans un projet Next.js. Firebase n'a rien à faire dans un projet Flutter. Moins de MCP globaux = moins de RAM par session = moins de dégâts en cas de fuite.

**3. Méfie-toi des hooks auto.** Un hook `session_start` qui spawn un processus lourd doit impérativement nettoyer les instances précédentes avant d'en créer de nouvelles. Sinon, le premier crash déclenche une cascade.

<Callout type="info" label="TL;DR">
Si ton Mac crash en boucle avec Claude Code ouvert → `ps aux | grep claude | wc -l`. Si le nombre est anormal, cherche quel plugin spawne des subagents orphelins, kill-les, et désinstalle le coupable.
</Callout>
